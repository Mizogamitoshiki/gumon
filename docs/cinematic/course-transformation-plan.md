# Course Transformation Plan（Phase 20A）

作成: 2026-07-11 ／ 状態: **承認待ち（実装未着手）**
対象: `/menu/course` のみ。現状分析は `course-project-analysis.md`。

このページの目的は「相談できる → 店が対応してくれる → 電話すれば決められる」という
**安心感**を届けること。主 CTA は電話。没入（Dinner）でも即断（Lunch）でもない。

- コピーが既に誠実:「ご予算に応じて」「ご相談ください」— 捏造ゼロで相談ページとして成立する素地がある
- MenuHero / MenuBoard / RM 分岐 / 44px タップは実証済みの共通基盤

## 2. Creative Concept 3 案

※ 各案のタイトルは設計の方向性ラベルであり、そのままページへ載せるコピーではない（新コピー追加は本計画のスコープ外）。

### 案A: 一卓の相談から、夜を整える。

| 観点 | 内容 |
| --- | --- |
| Concept | 「決まっていなくていい。一本の電話から、店が夜を整える」— 相談そのものを商品として見せる |
| 想定訪問者 | 人数・予算・内容が固まっていない幹事／記念日を任せたい人 |
| 起こしたい感情 | 敷居の低さ → 「この店は受け止めてくれる」という安心 → 電話への踏ん切り |
| 情報の順番 | ①ここは相談の入口（Hero）②対応できる 2 つの形＋飲み放題・貸切・人数予算の事実（紙カード＋notes）③電話で決められる（CTA） |
| 電話へ進む理由 | 「決めてから電話」ではなく「電話すると決まる」と提示されるため、未確定のままでも行動できる |
| Dinner 差別化 | Dinner=一皿と対面する没入。Course=対面の前の「約束を取り付ける」静かなページ |
| Lunch 差別化 | Lunch=速い即断。Course=急がせない。ゆっくり読める密度と間 |
| 実装コスト | 小（ギャラリー除去＋弱化 variant のみ。構造・コピー不変） |
| テンプレ感リスク | 低〜中。「相談ください」だけの寂しいページに見えるリスクは、紙カードの品と余白の設計で受ける |

### 案B: 人数と予算に合わせて、席をつくる。

| 観点 | 内容 |
| --- | --- |
| Concept | 対応条件（人数・予算・飲み放題・貸切）を情報として先に開示し、機能で信頼させる |
| 想定訪問者 | 条件が先に決まっている幹事（会社宴会など）。比較検討モード |
| 起こしたい感情 | 「条件が合う」という確認 → 納得 → 電話 |
| 情報の順番 | ①条件の一覧性（表・箇条書き的）②形の紹介 ③CTA |
| 電話へ進む理由 | 条件合致の確認が済んでいるため、電話が「最終確認」になる |
| Dinner 差別化 | 明確（情報 vs 没入） |
| Lunch 差別化 | **弱い**。「速く確認して決める」が Lunch の brisk と役割が近接する |
| 実装コスト | 中。条件の一覧化には**未確認情報（人数条件・飲み放題内容）が必須** — 現時点では作れない |
| テンプレ感リスク | **高**。「宴会コース対応表」はどの居酒屋サイトにもある型。愚問の声（問い）が消える |

### 案C: 特別な日も、いつもの集まりも。

| 観点 | 内容 |
| --- | --- |
| Concept | 利用シーンの幅（記念日〜気軽な集まり）を見せて間口を最大化する |
| 想定訪問者 | 記念日・接待・法事・同窓会など多様 |
| 起こしたい感情 | 「自分の用途もあり得る」という共感 → 電話 |
| 情報の順番 | ①シーン列挙 ②形の紹介 ③CTA |
| 電話へ進む理由 | 自分ごと化による後押し |
| Dinner 差別化 | あり（シーン訴求 vs 一皿訴求） |
| Lunch 差別化 | あり |
| 実装コスト | 中〜大。**シーン列挙は新コピーの追加が前提**（記念日対応可否など未確認情報にも接触しやすい） |
| テンプレ感リスク | **最も高い**。「歓送迎会・記念日・接待に」は業種テンプレの定番文言。Originality 原則と衝突 |

### 推奨: **案A**

理由: ①依頼文の目的（相談できる→対応してくれる→電話で決められる）と感情の並びが一対一で対応する
②既存コピー「ご予算に応じて」「ご相談ください」を**変更ゼロで**そのまま核に使える（捏造・文言承認が不要）
③案B は未確認情報が必須なうえ Lunch と役割が近接し、案C は新コピー前提でテンプレ感リスクが最大
④案B の美点（条件の事実開示）は、既存 notes（飲み放題・人数予算対応）の範囲内で C2 に取り込める。

## 3. Scene Map（3 Scene・演出強度は低く）

```text

C2 対応できることを知る … 紙カード2枚+notes が主役。空ギャラリーを除去し、Hero の次に静かに現す
C3 電話で席を整える   … CTA。読み終えた安心をそのまま電話へ
```

| | C1 相談の入口 | C2 対応できることを知る | C3 電話で席を整える |
| --- | --- | --- | --- |

## 4. Scroll Rhythm

```text
C1 中（入口の一拍: 文字せり上げ）→ C2 低（読む・信頼の区間 ≒ ページの大半）→ C3 低（静かな着地）
強度種別: 期待(C1) → 集中(C2) → 解放・余韻(C3)
```

- ピークは C1 の一度だけ。以降はページ末尾まで低強度を保つ（**低強度 ≥ 全体の 2/3**）
- pin・横スクロール・動画・スクロールジャック: **ゼロ**
- Hero→ボード間の余白は「間」として 1 箇所だけ設計（brisk の余白圧縮とは逆に、詰めすぎない）[推定: 実装時に実画面で微調整]

## 5. Motion 方針

| Motion | 対象 | 理由（憲章 7.4） | 削除条件 |
| --- | --- | --- | --- |
| 文字マスクせり上げ＋視差（既存） | C1 Hero | Scene の入口を作り、ブランドの世界観を保つ | MenuHero 共通仕様のため単独削除しない |
| fade-quiet 基準の弱い Reveal | C2 額縁・カード・notes | 読む区間で内容理解を妨げない最弱の登場（CDE 6.4: 読む区間は演出最弱） | RM/モバイルでは最初から構築しない |
| 既定の静かな CTA 登場 | C3 | 信頼→行動の受け渡し。主ボタンは演出対象外（常時予約可能を優先する既存原則） | — |

- トークンは `GUMON_SCENE_MOTION.fadeQuiet`（y8・短め・power1.out）を使用。**新トークン・新数値体系を作らない**
- Dinner の quiet「活気」（額縁躍り上がり y40・波状連鎖）と Lunch の brisk「軽快」（y28・速い連鎖）は**持ち込まない** — Course は「静けさ」。同じ分岐機構で値だけ最弱にする
