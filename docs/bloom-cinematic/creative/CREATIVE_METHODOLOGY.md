# CREATIVE_METHODOLOGY — Bloom-L Creative制作プロセス

本書は、Bloom-Lが案件開始から納品までに実行する思考と工程の方法論である。
Claudeはすべての案件で、毎回、本書の工程に従って思考する。新任のディレクターは、本書だけで制作フローの全体を理解できることを目指して書かれている。

## 本書の位置づけ

- 文書の優先順位における位置: **第4位（Phaseの設計書）**。上位は [PROJECT_CHARTER.md](../PROJECT_CHARTER.md) → [QUALITY_STANDARD.md](../QUALITY_STANDARD.md) → [CLAUDE.md](../BLOOM_RULES.md)
- 3つのcreative文書の役割分担:
  - [CREATIVE_DIRECTOR_ENGINE.md](./CREATIVE_DIRECTOR_ENGINE.md)（CDE） = **思考のOS**。どう考え、どう判断するか
  - [KNOWLEDGE_SYSTEM.md](./KNOWLEDGE_SYSTEM.md)（KS） = **百科事典**。何を知っていて、いつ引くか
  - 本書（Methodology） = **工程**。何を、どの順で、どの成果物として作り、どこで承認を得るか
- 本書はCDE 2章の思考フロー（Phase A〜D）を工程として展開したものである。矛盾する場合はCDEが正であり、停止して報告する

## 本書全体を貫く5つの工程原則

1. **上流から下流へ。** ブランド理解 → 体験 → 感情 → Story → Scene → Rhythm → Motion → 実装。下流から始めた仕事はすべて無効である（Brand Before Design / Story Before Motion — 憲章7.1・7.3）
2. **成果物駆動。** 各ステージは「考えた」では終わらない。定義された成果物が書けて初めて完了である。書けないことは、まだ考えられていないことである
3. **ゲート駆動。** 3つの承認ゲート（設計承認・代表Scene合格・納品判定）を通らずに次へ進むことはできない（憲章12章 段階的Creative Review）
4. **戻る勇気。** 下流の問題は一段上流で直す（CDE 2.4）。工程を戻ることは遅延ではなく、最短経路である
5. **記録が資産。** 判断・理由・測定値・未確認事項をすべて記録する。記録のない判断は、次の案件に何も残さない（憲章3章: プロセスの資産化）

### 1.2 最初の10分 — 「寿司屋です」と言われた瞬間に何をするか

依頼を受けた瞬間のClaudeの正しい動きを、例で固定する。

**やってはいけないこと**: Heroのビジュアルを想像する。「和モダンで、墨色で、筆のアクセントで」と考える。GSAPのタイムラインを考える。参考サイトを思い浮かべる。
頭に浮かんだそれらは、**他人のサイトの記憶**である（CDE 10章-1）。捨てる。

**やること（S0の実行）**:

1. 依頼の目的を確認する — 新規か変換か。何のためのサイトか（予約を増やす、単価を上げる、採用、周年）
2. 手元にある材料を棚卸しする — 既存サイト、写真、メニュー、クライアントの言葉、店舗の実在情報
3. 不明点をすべて列挙する — この時点で埋まっていないことを`[未確認]`として書き出す。**推測で埋めた瞬間、以降の全工程が汚染される**（CDE 9.4）
4. 「この店は何によって寿司屋なのか」をまだ知らない、と自覚する — 江戸前の仕事か、産地か、大将の人柄か、価格帯か、場所か。答えはS2で店から得るものであり、いま考えるものではない

つまり最初の10分の成果物は、ビジュアルではなく**要求整理メモ（目的・材料・不明点）**である。

### 1.3 各Stageの完了条件

各Stageの完了条件は本書2〜9章とCDE 2.3のゲート表に定める。共通条件は次の2つ。

- 成果物が定義どおりに書かれている（空欄・`[未確認]`の放置がない、または放置の理由が記録されている）
- 成果物が上位文書・上流成果物と矛盾していない

# 2. ブランド理解 — S2の方法論

分析の枠組み（8軸・確からしさラベル・翻訳規則）はCDE 3章が正である。本章は**実行の順番と手続き**を定める。

### 2.1 理解の順番

順番には理由がある。逆にすると、後の情報が先の情報に汚染される。

```
Step 1: 実物に触れる
        既存サイト・写真・商品・メニュー・店舗情報を、判断せずに観察する。
        この段階で「良い/悪い」「使える/使えない」を言わない。観察と評価を分ける

Step 2: 事実を並べる
        歴史・場所・価格帯・提供物・体制など、確認可能な事実を列挙する。
        各項目に[確認済み]/[推定]/[未確認]を付ける

Step 3: 言葉を聞く
        クライアント自身の言葉（既存の文章・過去の発信・ヒアリング済み内容）を
        原文のまま集める。要約しない。言葉の選び方自体が世界観の証拠である

Step 4: 8軸で仮説を立てる
        CDE 3.2の8軸（世界観・価値・差別化・歴史・ストーリー・文化・空気感・
        信頼の源泉）ごとに仮説を書く。全記述に確からしさラベルを付ける

Step 5: 競合と突き合わせる
        「競合のBrand Documentに書いたら嘘になるか」テスト（CDE 3.1）を
        全記述に実行する。嘘にならない記述は、まだ一般論である。掘り直す

Step 6: Interviewで埋める
        表現判断に影響する[未確認]だけを質問にする（下記2.2）

Step 7: Brand Documentとして確定する
        8軸＋「避けたい印象」＋固有の言葉集。以降の全工程の根拠になる
```

### 2.2 Interviewの設計方法

機械的な全問投げは禁止（憲章8章）。Interviewは**仮説検証型**で設計する。

1. **聞かないことを先に決める** — 解析で埋まった項目は聞かない。聞くのは「表現判断に影響する`[未確認]`」だけ
2. **開いた質問より、仮説をぶつける質問** — 「お店のこだわりは？」（開いた質問）は答えにくく、一般論が返る。「サイトの写真では、白木のカウンターが一番大切な場所に見えました。合っていますか？」（仮説検証）は具体が返る
3. **答えやすい順に並べる** — 事実確認 → 好み・嫌い（避けたい印象）→ 核心（なぜこの商売をしているのか）。核心をいきなり聞かない
4. **言葉を録る** — 回答は要約せず、クライアントの言い回しのまま記録する。その言い回しがコピーの声（QUALITY_STANDARD 17章）の原料になる

### 2.3 つまずきの兆候と対処

| 兆候 | 意味 | 対処 |
| --- | --- | --- |
| 8軸の記述が競合にも当てはまる | 一般論で埋めている | Step 1〜3に戻り、固有の証拠を探す |
| トーン（KS 1章）が即決できてしまう | 辞書引きデザインの入口 | トーンは分析の**後**。KS 掟1を読み直す |
| Interviewの質問が10個を超える | 解析をサボっている | 解析で埋まる質問を削る |
| 「避けたい印象」が空欄 | 重大な地雷が未検知 | 必ず聞く（QUALITY_STANDARD 1章 [Major]） |

# 3. Experience Design — S4の方法論

体験の定義（感情→記憶→信頼→行動の連鎖、三層、再生ヘッド）はCDE 4章が正である。本章はS4で作る3つの成果物の作り方を定める。

### 3.1 成果物1: 体験ゴール定義（3行）

訪問後の訪問者の状態を、3つの文で定義する。

```
感情: 訪問者は ________ という感情の変化を経験している
記憶: 閉じた後、________ が残っている（場面1つ・言葉1つまで具体に）
行動: その結果、________ する（予約する・保存する・人に話す 等）
```

**検査**: この3行を競合サイトのゴールとして書いたら嘘になるか。ならないなら書き直し。「良い店だと思ってもらう」は全店で書けるのでゴールではない。

### 3.2 成果物2: 記憶層の選定

ページ全体で**記憶に残す1〜2箇所**を、この段階で決める（CDE 4.4）。Scene設計より先に決めるのは、記憶層が決まっていないと全Sceneを均等に磨いてしまい、結果としてどこも記憶に残らないからである（ピークエンドの法則 — KS 7.6）。

選定基準: ①ブランドの核（Brand Documentの最も固有な記述）を担えるか ②体験ゴールの「記憶」の行と一致するか ③圧倒または感動を担えるか。

### 3.3 成果物3: コンセプトステートメント（1〜3行）

ページ全体のディレクション方針を1〜3行で書く（QUALITY_STANDARD 2章 [Blocker]）。全Sceneはこの文に従い、逸脱したSceneはレビューで検出される。

**書き方**: 「〜のようなサイト」（比喩・参照）で書かない。**訪問者の中で起きること**で書く。
- 悪い例: 「高級旅館のような落ち着いたサイト」（参照依存・一般論）
- 良い型: 「入店から会計までの2時間を、スクロールの3分に凝縮する。訪問者は席に着く前に、もう大将の間合いを知っている」

# 4. Emotion Design — S5の方法論

感情の語彙・文法・遷移表はKS 2章が正である。本章は感情曲線の**作図手順**を定める。

### 4.1 感情曲線の設計手順

```
Step 1: 出口から決める
        体験ゴールの「感情」の行が、曲線の終点である。終点を先に打つ

Step 2: 入口を決める
        訪問者理解（S3）から、訪問者が来た瞬間の状態を書く
        （例: 検索で流入、比較中、疑い半分、モバイル、移動中）。
        曲線は訪問者の現在地から始まる。ブランドが始めたい地点からではない

Step 3: ピークを配置する
        記憶層（3.2）と対応する位置に、ピーク（圧倒・感動など）を置く。
        ピークは1〜2個（KS 2.3: 圧倒は1ページ1〜2回）

Step 4: 静けさを配置する
        各ピークの直後に受け止めの区間を置く（QUALITY_STANDARD 7章 [Blocker]）。
        ピークと静けさは必ず対で設計する

Step 5: 間を埋める
        入口→ピーク→出口をつなぐ感情を、KS 2章の遷移表で検査しながら埋める。
        遷移表にない接続（例: 高揚→静寂の直結）が必要なら、間に感情を足す

Step 6: 全体を読み上げて検査する
        曲線を物語のように音読する。
        「疑い半分で来て、迎え入れられて安心し、間合いに引き込まれて期待が募り、
        一枚の写真に圧倒され、静けさで受け止め、余韻の中で席を予約する」
        — 読み上げて不自然な箇所が、設計の穴である
```

### 4.2 感情曲線の記法

```
位置(%)   0      15      35      55      65      85      100
感情      警戒    安心    期待    圧倒    静寂    幸福    余韻→行動
根拠      S3訪問者定義  Brand Doc「もてなし」  ...（各点に根拠を付す）
```

記法は自由だが、①各感情点に根拠（Brand Document・訪問者定義への参照）②ピーク≥1・静けさ≥1（QUALITY_STANDARD 3章）を必須とする。

### 4.3 つまずきの兆候と対処

| 兆候 | 意味 | 対処 |
| --- | --- | --- |
| 全区間が「すごい」系 | 感情が平坦に張り付いている（QS 3章 不合格例） | Step 4からやり直す。静けさは失敗ではない |
| 感情に根拠が書けない | ブランド・訪問者と接続していない | S2/S3へ戻る |
| ピークが3つ以上 | 記憶層の未選定か優柔不断 | 3.2に戻り1〜2個に絞る |
| 入口が「ワクワク」から始まる | 訪問者の現在地を無視した願望 | Step 2をやり直す |

# 5. Story Design — S6の方法論

### 5.1 Storyとは何か

Storyとは、**情報開示の順番に必然性を与える設計**である（QUALITY_STANDARD 4章）。飾りの物語文を書くことではない。「なぜこれが先で、あれが後なのか」に全て答えられる状態を作ることである。

### 5.2 Story設計手順

```
Step 1: 一文で書く
        このページの物語を1〜2文で書く（一文テスト — QS 4章）。
        主語は訪問者かブランドの核。
        例の型: 「◯◯だと思って来た訪問者が、△△を知り、□□に変わる物語」

Step 2: 感情曲線と重ねる
        S5の曲線の各区間に、物語の節（起・承・転・結、または序破急）を対応させる。
        感情曲線が「どう感じるか」、Storyが「何を知るか」。2本は常に並走する

Step 3: 開示順を決める
        伝えるべき情報を全て列挙し、「訪問者がその情報を受け取れる状態になる順」
        に並べる。価格は信頼の後。予約は物語の結末（Trust Before Action — 憲章7.7）。
        各順序に理由を1行ずつ記録する

Step 4: 捨てる
        物語の節に割り当てられない情報は、このページから外す（フッター・別ページへ）。
        物語の途中に割り込む無関係ブロックは、Storyを殺す（QS 4章 不合格例）

Step 5: 接続を検査する
        起点（出会い）と結末（CTA）が一本の線でつながっているか。
        結末から逆に読んで、各節が結末の必然性を積んでいるか
```

### 5.3 開示順の定石と、その破り方

定石: **世界観 → 具体（証拠）→ 人 → 実用情報 → 行動**。信頼は「世界観に共感 → 証拠で確信 → 人で安心」の順に積み上がる（KS 7.4 信頼の4源泉）。

ただし定石は一般解である。Brand Documentが逆を要求するなら破る（例: 「価格の潔さこそがこの店の思想」なら、価格を冒頭に置く判断があり得る）。破る場合は理由を記録する。破った理由が書けないなら、破ってはいけない。

# 6. Scene Design — S7の方法論

### 6.1 Sceneとは何か

Sceneとは、**1つの目的（伝える情報＋起こす感情）を持つ体験の単位**である（QUALITY_STANDARD 5章）。セクション（情報の区切り）ではない。感情の区切りである。

- セクション思考: 「メニュー / こだわり / 店舗情報 / アクセス」— 情報の種類で切る
- Scene思考: 「静寂の入店 / 手元への接近 / 大将の一言 / 席への案内」— 感情が変わる場所で切る

### 6.2 Scene設計手順

```
Step 1: 感情曲線とStoryを重ねて、切れ目を見つける
        感情が変わる場所・物語の節が変わる場所が、Sceneの境界である

Step 2: 各SceneにScene定義シートを書く（6.3）

Step 3: 重複を統合する
        目的が同じSceneが2つあれば統合または削除（QS 5章 [Major]）

Step 4: フックを設計する
        各Sceneに「次を見たい」の装置を1つ選ぶ（KS 6.4 好奇心カタログ）。
        同一手法の連続≤2。カタログ全種を使うことは目的ではない（KS 8.5）

Step 5: 記憶層Sceneを特定する
        3.2で選んだ記憶層がどのSceneかを明記し、そのSceneには
        他Sceneより設計密度を集中させる
```

### 6.3 Scene定義シート — 全Sceneに必須

```
Scene名:       （設計書と実装で同名にする — QS 19章）
目的:          伝える情報1つ＋起こす感情1つ
物語上の節:    Storyのどの節を担うか
感情曲線の区間: どの感情からどの感情への区間か
フック:        次のSceneへの好奇心装置（手法名）
主役:          このSceneで最初に目が行くべき1要素（QS 12章）
強度:          高/中/低（S8のRhythm Scoreに引き継ぐ）
根拠:          Brand Documentの参照箇所
```

空欄のあるSceneは実装に進めない。埋められない欄は、そのSceneが不要である兆候として扱う（迷ったら減らす — 憲章12章-4）。

### 6.4 Sceneの数

適正数は物語が決める。目安として、1Sceneに5つ以上のメッセージが詰まれば分割し（QS 5章）、物語の節に割り当てられないSceneは削除する。「豪華にするためにSceneを足す」は演出の詰め込みであり禁止。

# 7. Scroll Rhythm — S8の方法論

Rhythm理論（呼吸モデル・6強度語彙・譜面の規則）はCDE 7章が正である。本章は譜面の**作成手順**を定める。

### 7.1 Rhythm Score作成手順

```
Step 1: Scene一覧の「強度」列を並べる（S7から引き継ぐ）

Step 2: 各Sceneに強度語彙を割り当てる
        感情曲線の感情（目的）を、6つの強度語彙（手段）に翻訳する:
        静けさ・期待・インパクト・集中・解放・余韻（QS 7章と同一語彙）

Step 3: 呼吸を記す
        各Sceneに吸う/止める/吐くを記入し、呼吸の連なりとして読む（CDE 7.1）

Step 4: 規則を検査する
        - 高強度の連続 ≤ 2
        - 低強度区間 ≥ 全体の20%目安（変更時は理由を記録）
        - インパクトの直後は必ず「吐く」
        - 冒頭は約束、末尾は余韻→行動（CDE 7.4）

Step 5: テンポを重ねる
        読む区間（集中）は密度高く、感じる区間は距離に余裕を（KS 6.3）。
        Sceneの高さの目安をここで決める

Step 6: 単調を検査する
        呼吸パターンが機械的な反復になっていないか。
        反復するならモチーフとして変奏されているか（CDE 7.4-7）
```

### 7.2 「静 → 動 → 静 → 驚き → 余韻」を設計するとは

この並びは譜面の一例である。設計の実体は次の対応にある。

| 区間 | 呼吸 | 何をするか |
| --- | --- | --- |
| 静 | 吐く/吸う | 世界観への入場。情報を絞り、訪問者の速度を落とさせる |
| 動 | 浅く止める | 最初の展開。世界が動き出す。ただし最大出力は出さない |
| 静 | 吐く | 一度受け止めさせ、次の「吸う」を仕込む（見切れ・予告 — KS 6.4） |
| 驚き | 止める | ピーク。事前に作った予測を世界観の内側で裏切る（KS 6.5）。記憶層の本命 |
| 余韻 | 深く吐く | 減衰と沈黙。体験を記憶に定着させ、行動（CTA）へ橋を架ける |

**警告**: この5区間をそのまま全案件に適用した瞬間、それはテンプレートである（憲章7.8違反）。譜面は感情曲線から毎回導く。この表は「並びの読み方」の教材であり、答えではない。

### 7.3 つまずきの兆候と対処

| 兆候 | 意味 | 対処 |
| --- | --- | --- |
| 全Sceneが「中」強度 | 緩急の未設計（一様は不合格 — 憲章7.6） | Step 2からやり直す |
| 低強度区間を「もったいない」と感じる | 静けさへの不安 | 静けさは設計要素（憲章12章-4）。KS 2.4を読む |
| 譜面が前案件と似る | エンジンの自己模倣（憲章11章 量産感の再発） | 感情曲線から導き直す。似た理由を説明できなければ棄却 |

# 8. Motion Decision — S9の方法論

判断基準（採用4条件・削除6条件・弱化5条件・強化4条件・強度4変数）はCDE 6章が正である。本章は**工程としての実行方法**を定める。

### 8.1 Motion設計の手順

```
Step 1: Motion一覧表を先に作る
        Scene一覧の全Sceneについて、候補Motionを列挙する。
        実装しながら思いつきで足すことを禁止する（表にないMotionは実装しない）

Step 2: 各候補をCDE 9.2の判定にかける
        理由（憲章7.4リスト）→ 下限 → 同型検査 → 強度決定 の順。
        1つでも通らなければ表から消す

Step 3: トークンを定義する
        採用Motionのduration・easing・距離を、場当たりでなく
        トークンセットとして定義する（QS 8章 [Major]）。
        トークンの水準はブランドトーン5軸から導き、Brand Documentで上書きする

Step 4: reduced-motion代替を同時に設計する
        後付けにしない。代替が設計できないMotionは、この段階で消す
        （QS 14章: reduced-motion時の全コンテンツ表示は[Blocker]）

Step 5: 削除テストの計画を書く
        どのMotionを、いつ、どう無効化して比較するかをQA計画に記す
```

### 8.2 Motion一覧表 — 全Motionに必須

```
対象Scene / 対象要素:
技法:          （KS 5章の技法名）
理由:          憲章7.4のどれに当たるか＋このMotion固有の言葉で1行
強度:          トークン参照（距離・duration・easing・同時性）
reduced-motion代替:
削除テスト結果: （実装後に記入）
```

理由欄が「雰囲気が良くなる」「他で見て良かった」の類なら、その行は即削除する。削除に承認は不要（CDE 1.4）。

### 8.3 採用・不採用・削除の運用整理

- **採用する**: CDE 6.2の4条件をすべて満たす。採用は常に「表に書けた」ことを意味する
- **採用しない（設計段階で見送る）**: 条件を満たさない、または満たすが**なくても体験が成立する**。迷ったら見送る（憲章12章-4: 迷ったら減らす）。見送りの記録は残す — 次案件の判断材料になる
- **削除する（実装後に消す）**: 削除テストで「無い方が良い/変わらない」、実機で読解・操作・性能を害した、reduced-motion代替が実装として成立しなかった。実装済みであることは削除を躊躇する理由にならない — **書いたコードへの愛着は判断基準ではない**

# 9. Creative Review — 各工程のレビュー方法論

レビューの原則（3視点・4パス・出力フォーマット）はCDE 8章が正である。本章は**どの工程で・何を・どうレビューするか**を定める。

### 9.1 工程内レビュー（各Stageの出口で、Claudeが自己実行）

| Stage | レビュー内容 |
| --- | --- |
| S2 | 競合テスト全記述実行。`[未確認]`の放置チェック。「避けたい印象」の記載確認 |
| S4 | 体験ゴール3行の競合テスト。記憶層1〜2箇所の確定。コンセプトが参照依存でないか |
| S5 | ピーク≥1・静けさ≥1。全感情点の根拠。遷移表（KS 2章）違反の検出 |
| S6 | 一文テスト。全Sceneの節割り当て。開示順理由の記録 |
| S7 | Scene定義シートの空欄ゼロ。目的重複ゼロ。フック連続≤2 |
| S8 | 譜面規則4項目（高強度連続・低強度比率・受け止め・冒頭末尾） |
| S9 | Motion一覧の理由記録率100%。トークン定義。reduced-motion代替の設計済み |

### 9.2 G1: 設計承認（人間承認 — 必須ゲート）

- **提出物**: S2〜S9の全成果物＋Claudeの推奨と理由（丸投げ禁止 — CDE 1.2）
- **レビュー方法**: CDE 8.3のPass 2（原則パス）とPass 3のうち設計段階で実行可能な項目（一文テスト・Scene表・譜面照合）
- **不利な材料を先に**: `[未確認]`の一覧、自信のない判断、ボーダーラインを明示的に提出する

### 9.3 G2: 代表Scene Creative Review（人間判定 — 必須ゲート）

- **対象**: Heroまたは記憶層Scene（世界観の実在を最も検証できるScene）
- **レビュー方法**: 代表Sceneに対するCDE 8.3の全4パス（体験・原則・検証・下限）。実機・reduced-motion・375px幅を含む
- **判定の意味**: ここでの不合格は安価、全面実装後の不合格は高価（CDE 8.6）。不合格なら該当Stageへ戻る — 大抵はS9ではなくS4〜S7に原因がある（CDE 2.4 バックトラック）

### 9.4 S12: 三段QA

1. **Technical QA**（QUALITY_STANDARD 22章）— 最低セット（a11y自動チェック・Lighthouseモバイル・reduced-motion・CPUスロットリング・対象ブラウザ・実機・CTA動作）を先に実施。技術的に壊れた状態でCreative QAをしても意味がない（QS 0章）
2. **Creative QA**（QS 21章）— A〜C章全[Blocker]の判定記録、通しスクロール（PC・モバイル）、第三者テスト、疲労チェック、量産感チェック
3. **Final Creative QA** — 修正後の最終通し確認（憲章12章）。修正が影響した項目のみ再QAし、全体を最後にもう一度通しで見る

### 9.5 G3: 納品判定（人間判定 — 必須ゲート）

- 全[Blocker]合格＋[Major]不合格の承認記録が揃って初めて判定に進める（QS 0章）
- パフォーマンスは**ラボ合格・フィールド未計測**と正確に報告する（QS 16章: 未計測のフィールドデータを合格済みとして扱わない）

### 9.6 S13: 公開後運用QA

- CrUXまたはRUMでLCP・CLS・INPのフィールド値を確認し、運用QAとして記録する（QS 16章）
- 案件の学び（上書きが起きた点・仮説の当否・目安値の妥当性）を各設計書の改訂提案として記録する。**これが2案件目の品質の原資である**（憲章10章: プロセスの再利用）

# 10. Decision Tree — 意思決定の擬似コード

CDE 9章（メインループ・Motion判定・衝突解決・whyChain）が思考の擬似コードの正である。本章は**工程の進行判断**をDecision Treeとして定める。

### 10.1 工程進行の全体Tree

```
function runProject(request):

    # ===== S0: 受理 =====
    memo = organize(request)                    # 目的・材料・不明点
    if memo.purpose is unclear:
        ask(クライアントの目的)                   # 目的不明のまま進まない
    forbid(visualIdeation)                      # この時点のビジュアル着想は全て破棄

    # ===== S1〜S3: 理解 =====
    analysis = analyze(existingSite, materials, competitors, techStack)
    brand    = buildBrandDocument(analysis)      # 本書2章の7 Step
    if brand.failsCompetitorTest():  goto S2.Step1   # 一般論なら掘り直し
    visitor  = defineVisitor(analysis, brand)

    # ===== S4〜S8: 設計 =====
    goal     = defineExperienceGoal(brand, visitor)         # 3行 + 競合テスト
    memory   = selectMemoryScenes(goal, brand, max=2)       # Sceneより先に決める
    concept  = writeConceptStatement(goal)                  # 参照依存なら書き直し
    emotion  = drawEmotionCurve(goal, visitor, memory)      # 本書4章の6 Step
    story    = designStory(brand, emotion)                  # 一文テスト + 開示順
    scenes   = designScenes(emotion, story)                 # 定義シート全欄必須
    rhythm   = writeRhythmScore(scenes)                     # 本書7章の6 Step

    # ===== 中間検査: 上流へ戻る判断 =====
    if scenes.cannotFill(定義シート):    return to S6 (Storyの多重化を疑う)
    if rhythm.looksLike(過去案件):       return to S5 (感情曲線から導き直す)
    if noOverrideOccurred(KS一般解):     return to S2 (ブランド理解が浅い — CDE 9.6)

    # ===== S9: 表現 =====
    motions  = buildMotionTable(scenes, rhythm, brand)      # 表にないMotionは存在しない
    for m in motions: decideMotion(m)                       # CDE 9.2に委譲
    tokens   = defineTokens(brand.toneAxes, brand.overrides)

    # ===== G1: 設計承認 =====
    verdict = humanReview(allArtifacts, claudeRecommendation, unfavorableFirst)
    if verdict != approved: backtrack(verdict.reason); goto 該当Stage

    # ===== S10〜G2: 代表Scene =====
    hero = implement(representative(memory, scenes))
    verdict = humanReview(fourPassReview(hero))             # CDE 8.3
    if verdict != approved: backtrack(mostUpstreamCause(verdict))   # S9ではなくS4〜S7を先に疑う

    # ===== S11〜S12: 全面実装とQA =====
    for scene in remaining(scenes):
        implement(scene, stepwise=true)                     # 大規模一括書き換え禁止
    run(technicalQA)                                        # 先
    if technicalQA.hasBlocker: fix(); rerun(affectedOnly)
    run(creativeQA)                                         # 後
    run(finalCreativeQA)

    # ===== G3: 納品判定 =====
    report(blockers=all判定, unverified=明示, field="未計測")
    verdict = humanDeliveryDecision()

    # ===== S13: 公開後 =====
    schedule(fieldDataCheck)                                # CrUX/RUM → 運用QA記録
    harvest(learnings → 各設計書への改訂提案)
```

### 10.2 停止条件 — 進行より優先される判断

以下を検出したら、工程の進行を止める。

```
if 文書間の矛盾を発見:            stop(); report(箇所・影響・推奨修正)   # 憲章冒頭
if 下限(a11y/reduced-motion/モバイル/低性能/ライセンス)と演出が衝突:
                                  下限を守り、演出を再設計。下限側は動かさない
if 大規模変更・リプレイスが必要と判断:
                                  実行せず、理由とともに提案し承認を待つ  # CLAUDE.md
if [未確認]を確認済みとして書きそうになった:
                                  書かない。ラベルを付けて質問へ回す      # CDE 9.4
```

### 10.3 優先順位の解決

判断が衝突したら、憲章12章の意思決定原則を上から順に適用する（下限 → 訪問者の目的 → ブランド適合 → 減らす → 測定と根拠 → 小さく検証 → 不確実性の明示）。この順序は本書のいかなる工程判断よりも強い。

### 本書の前提と仮定

- 工程の粒度（S0〜S13）はv1（愚門）を想定した初期設計である。実案件検証後に統合・分割を見直す
- Interview手法（2.2）・感情曲線記法（4.2）・各シート様式は初期仮説であり、運用で改良する。様式の変更は自由だが、必須項目（根拠欄・ラベル・規則検査）の削除は本書の改訂として扱う
- Agent・Skill・Command・`/storytelling`への工程の割り当ては、Architecture Phase以降で設計する。本書は「誰が（どのAgentが）やるか」を定めない

### 改訂

実案件での検証結果か、上位文書（憲章・QUALITY_STANDARD・CDE）の変更のみを改訂の理由とする。工程の追加は「その工程がないことで実際に品質事故が起きた」証拠とともに提案する。工程は増やすほど良いものではない — 工程にもRestraint（憲章7.9）が適用される。
