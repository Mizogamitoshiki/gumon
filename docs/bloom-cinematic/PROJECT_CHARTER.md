## 1. Mission

クライアントのブランド・想い・価値観を理解し、それを記憶に残るスクロール体験へ変換する。

Bloom-L Cinematic Engineは、そのための制作エンジンである。既存または新規のクライアントWebサイトを解析し、
ブランド固有の感情設計・ストーリー・Scene構成・Motionを設計・実装し、
「思わず最後までスクロールしたくなる」体験としてクライアントに納品できる状態を作る。

## 2. Vision

Bloom-Lは、ホームページを制作する会社ではなく、ブランドを体験へ変換するクリエイティブパートナーである。

5年後に目指す状態:

- 海外の著名なWebデザイナー・制作会社と比較されても見劣りしない品質を出せる
- スクロールするたびに映画のような演出を、ブランドごとに作り分けられる
- Claude Codeを制作速度の道具ではなく、品質・検証量・改善速度の増幅装置として使いこなしている
- 1案件100万円以上でも依頼したいと思われ、価格ではなく体験価値と品質で選ばれる

このVisionにおいて、エンジンの役割は「Bloom-Lの制作品質の下限を引き上げ、上限への挑戦を繰り返せる構造を作ること」である。

## 3. 背景

- 高品質なシネマティックWeb体験（Awwwards受賞級）は、通常、多人数のスタジオが長期間かけて作る。個人・少人数のBloom-Lが同じ土俵に立つには、制作プロセス自体を資産化する必要がある
- 生成AIによるサイト制作は普及したが、その多くは「テンプレートに情報を流し込んだ量産感のあるサイト」を生む。Bloom-Lはその逆、つまりAIを使うほどブランド固有性が深まる仕組みを目指す
- 案件ごとにゼロから設計をやり直すと、品質が担当者の記憶とその日の判断に依存する。原則・品質基準・プロセスを文書化し、エンジンとして再利用可能にすることで、品質の再現性を確保する

## 4. 解決する課題

| 課題 | エンジンによる解決 |
| --- | --- |
| ブランド理解が浅いままデザインが始まり、どの会社にも当てはまるサイトになる | 解析とCreative Interviewを実装より先に置くプロセスを強制する |
| 演出が目的化し、重い・疲れる・読めないサイトになる | Motion Has Meaning / Restraint / Scroll Rhythmを合否基準として運用する |
| 品質判断が主観的・属人的で、納品可否がぶれる | QUALITY_STANDARD.mdで合格条件・不合格例・測定基準を定義する |
| AI生成サイト特有の既視感・量産感 | Originality原則と量産感チェック（QUALITY_STANDARD 9章）で検出・排除する |
| 高品質演出とパフォーマンス・アクセシビリティの両立が難しい | 両立を「望ましい」ではなく交渉不可の合格条件として定義する |

## 5. 対象ユーザー

- **一次ユーザー**: Bloom-Lの制作者（当面は代表1名を想定）。Claude Code上でエンジンを実行し、設計判断と最終品質判断を行う
- **二次ユーザー**: Bloom-Lのクライアント（中小規模の店舗・企業。v1検証対象は飲食店「愚門」）。エンジンを直接操作しない
- **最終受益者**: クライアントサイトの訪問者。体験品質の最終判断基準はこの層に置く

エンジンは完全無人納品を目指さない。一次ユーザーの判断を置き換えるのではなく、判断の質と検証量を引き上げる。

## 6. 提供価値

Bloom-Lの価値は、AIやClaude Codeの使用そのものではなく、次の能力にある。

1. クライアントの本質（世界観・こだわり・空気感・他社との違い）を理解する
2. 訪問者に起こしたい感情の変化を設計する
3. 信頼が生まれる情報開示の流れを設計する
4. 「次を見たい」という好奇心を持続させる構成を設計する
5. スクロール全体の緩急と余韻を設計する
6. ブランドごとに異なる体験を作り、テンプレートを排する
7. 最新技術を目的ではなく表現手段として選択する
8. 世界中の最新Web表現を継続的に研究し、再利用可能な設計原理として抽出する

### 参考とする体験品質と、その扱い

WEDNES、Apple、Stripe、Active Theory、Locomotive、Awwwards掲載・受賞サイト、
ヤマトグループなどの高品質な企業周年・ブランドストーリーサイト等の没入型Web体験を研究対象とする。

研究対象から抽出してよいのは再利用可能な設計原理のみである:
没入感、スクロールのテンポ、パララックス、Scene構成、Transition、余白、タイポグラフィ、
情報開示の順番、動と静の対比、好奇心の持続方法、感情曲線、信頼形成、CTAまでの導線、パフォーマンスとの両立。

デザイン・構成・素材・コード・特徴的表現をそのまま模倣することは禁止する。
目的は参考サイトと同じものを作ることではなく、同等以上の体験品質をクライアント独自の表現として実現することである。

## 7. 設計原則

以下は暫定案ではなく、今後すべてのAgent、Skill、Command、設計書、実装、レビューが従う最上位原則である。

### 7.1 Brand Before Design

ブランドを理解せずに、デザインやモーションを決めてはならない。
最初に理解すべきものは、世界観、ブランドコンセプト、店舗や事業の空気感、商品・サービスへのこだわり、他社との違いである。

### 7.2 Emotion First Design

最初に「訪問者にどのような感情の変化を起こしたいか」を設計する。
デザイン、Story、Scene、Motionは、その感情を実現するために選択する。

### 7.3 Story Before Motion

Motionを先に選んではならない。ブランド理解、感情設計、Story、Scene構成を決めた後にMotionを選ぶ。

### 7.4 Motion Has Meaning

動かすこと自体を目的にしてはならない。すべての主要なMotionには、次のいずれかの理由が必要である。

- 内容を理解しやすくする
- 次への期待を生む
- 感情を変化させる
- ブランドの世界観を強める
- 情報の優先順位を示す
- Sceneを自然につなぐ
- 信頼または行動へ導く

理由を説明できないMotionは削除する。

### 7.5 Curiosity-Driven Scroll

各Sceneは、訪問者に「次に何が起きるのか」と思わせる役割を持つ。
ただし、毎回同じRevealや派手な演出を繰り返してはならない。
期待、驚き、理解、静けさ、余韻を組み合わせ、スクロールする理由を継続的に作る。

### 7.6 Scroll Rhythm

ページ全体を一様な速度、一様な密度、一様な派手さにしてはならない。
静けさ、期待、インパクト、集中、解放、余韻の緩急を設計する。
派手な演出の後には、内容を受け止められる静かな時間と余白を設ける。
見ていて疲れず、AIが演出を詰め込んだように見えないことを必須とする。

### 7.7 Trust Before Action

問い合わせや予約は、ボタンを目立たせるだけでは生まれない。
訪問者が「この会社・店舗なら任せられる」と感じ、信頼が十分に積み上がった結果としてCTAへ進む構成を作る。

### 7.8 Originality

案件ごとに、ブランド、感情曲線、Story、Scene、Rhythm、Motionの組み合わせを再設計する。
業種別テンプレートをそのまま当てはめてはならない。
パターンやライブラリ（部品）は再利用してよいが、完成した体験（組み合わせ）はクライアント固有でなければならない。

### 7.9 Restraint

すべてを動かさない。見せ場を成立させるために、動かない要素、静かなScene、大きな余白、短いコピーを意図的に使用する。
減らす判断を、追加する判断と同じくらい重視する。

### 7.10 Quality Over Speed

短期間で動くことより、Bloom-Lの名前でクライアントへ納品できる品質を優先する。
ただし、抽象的な理想論による過剰設計も避ける。小さく実装し、実案件で検証し、継続的に改善できる構造にする。

## 8. v1スコープ

v1の中心は以下とする。

- Claude Code上で動く
- 既存Webプロジェクトを解析できる
- Creative Interviewを必要に応じて行える（機械的な全問投げは禁止。既存情報から判断できる項目は解析で埋め、不足分だけを回答しやすい順で質問する）
- Brand、Emotion、Story、Scene、Transition、Rhythm、Motionの設計案を作れる
- 承認前に大規模変更を行わない。承認後に既存サイトを段階的に変換できる
- GSAP、ScrollTrigger、Lenisを適切に導入できる
- レスポンシブと`prefers-reduced-motion`に対応できる
- Creative QAと技術QAを実施できる
- 最初の検証対象として、既存の飲食店サイト「愚門」を使用する
- WEDNESのような没入感と好奇心を参考にしつつ、愚門独自の体験を作れる

### 技術方針

想定技術: React / Next.js、TypeScript、CSSまたは適切なスタイリング方式、GSAP、ScrollTrigger、Lenis、SVG、Canvas、Three.js / WebGL。

ただし全案件で固定使用しない。案件の目的、既存構成、必要な体験、保守性、パフォーマンスに応じて最小限の適切な技術を選ぶ。
Three.js、Canvas、WebGLは、体験価値を大きく高め、DOM・CSS・GSAPで代替しにくい場合のみ採用する。
既存案件の変換では現行スタックを尊重し、必要性のない全面リプレイスを行わない。

## 9. 非目標（v1で追わないもの）

- あらゆる業種・あらゆるフレームワークへの完全対応
- 100種類以上の完成済みMotion Library
- WebGLを必須とする表現
- 完全無人でのクライアント納品
- 1コマンドで確認なしに全コードを書き換えること
- Awwwards受賞の保証
- 特定参考サイトの完全複製
- 調査・設計・実装・QAを一度の巨大なプロンプトで完結させること

### 絶対に作らないもの（バージョンを問わない）

- テンプレート感の強いサイト、AIが短時間で量産したと感じるサイト
- 古いWordPressテーマのような既視感のあるサイト
- Heroだけが派手で、その後は普通になるサイト
- すべての要素が同じFade Upで表示されるサイト
- 動きが多すぎて情報を読めないサイト、強いスクロールジャックで操作を奪うサイト
- ブランドと無関係な流行演出を詰め込んだサイト
- 見た目だけで、問い合わせ・予約・信頼につながらないサイト
- モバイルで破綻するサイト、低性能端末で極端に重くなるサイト
- アクセシビリティや`prefers-reduced-motion`を無視したサイト
- 参考サイトのコピー、ライセンスや知的財産権を無視した実装
- 保守できない一枚岩のアニメーションコード、Claudeにしか理解できない構造

## 10. 成功条件

### v1の成功条件

1. 愚門サイトが、QUALITY_STANDARD.mdの全Blocker項目に合格した状態で変換されている
2. 変換の各段階（解析 → Interview → 感情設計 → Story → Scene → Transition → Rhythm → Motion → 実装 → QA）が、承認を挟みながら実行できた
3. 愚門の体験が、参考サイトの模倣ではなく愚門固有であると、量産感チェック（QUALITY_STANDARD 9章）で確認できる
4. 完成物を第三者（制作関係者以外）に見せたとき、「最後までスクロールした」「テンプレートに見えない」という反応が得られる
5. このプロセスが文書化され、2案件目に再利用できる状態になっている

### エンジンとしての中期成功条件

- 案件ごとの体験設計（感情曲線・Scene構成）が毎回異なるものになっている
- 品質判断の大半がQUALITY_STANDARD.mdへの参照で説明できる（「なんとなく良い/悪い」が減る）
- 研究した設計原理が、模倣ではなく部品・原則として蓄積されている

## 11. 主要なリスク

| リスク | 内容 | 対策 |
| --- | --- | --- |
| 量産感の再発 | エンジン自体がパターン化し、案件間で似た体験を量産する | 案件ごとの感情曲線・Scene構成の再設計を必須化。量産感チェックを Creative QA に組み込む |
| 演出過多 | 「シネマティック」の名の下に演出が目的化する | Motion Has Meaning / Restraint を合否基準化。削除テスト（消して劣化しなければ削除）を運用 |
| 品質とパフォーマンスの衝突 | 高品質演出が低性能端末・モバイルで破綻する | Core Web Vitals と低性能端末検証を Blocker 条件にする |
| 模倣の混入 | 参考研究が意図せずデザイン模倣になる | 抽出対象を「設計原理」に限定。Creative QA で参考サイトとの近似を確認 |
| 過剰設計 | 実案件前に抽象的な仕組みを作り込みすぎる | v1は愚門1案件での検証を完了条件とし、汎用化は実績後に行う |
| 属人性 | 品質の最終判断が代表1名の主観に留まる | 測定可能な基準を最大化し、人間判断が必要な項目は明示的にリスト化する |
| セッション間の一貫性 | Claudeセッションごとに判断がぶれる | CLAUDE.md・憲章・品質基準を毎セッションの参照必須とする |
| 法務 | フォント・写真・映像・コードのライセンス違反 | 素材・依存ライブラリのライセンス確認を Technical QA に含める |
| 検証の甘さ | 「動いた」を「完成」と誤認する | 完了の定義を QUALITY_STANDARD 合格に固定。検証結果の報告を必須化 |

## 12. 意思決定原則

判断に迷った場合、以下の順で決める。

1. **交渉不可の下限を先に守る** — アクセシビリティ、`prefers-reduced-motion`、モバイル動作、低性能端末での実用性、ライセンス遵守は、いかなる演出とも交換しない
2. **訪問者の目的達成 > ブランド体験 > 表現の新しさ** — 読めない・使えない演出は、どれほど美しくても不合格
3. **ブランド適合 > 一般的な良さ** — 「一般に良いとされる表現」でも、そのブランドに合わなければ採用しない
4. **迷ったら減らす** — 追加と削除で迷ったら削除する。静けさは失敗ではなく設計要素である
5. **測定と根拠 > 印象論** — 可能な限りQUALITY_STANDARD.mdの基準・測定値で議論する。人間判断が必要な項目は、判断者と判断理由を記録する
6. **小さく作って実案件で検証する** — 仕組みの汎用化は、実案件での検証後に行う
7. **不確実性を明示する** — 確認できていないことを確認済みのように扱わない

### 段階的Creative Review（v1プロセスの原則）

- 全面実装の前に、Brand・Emotion・Story・Scene・Rhythmの設計レビューを行う
- Heroまたは代表Sceneを先行実装し、そのCreative Reviewに合格してから全面実装へ進む
- 最終納品前は、Technical QAの後にFinal Creative QAを実施する

詳細なワークフロー設計はArchitecture Phaseで行う。ここでは原則のみを定める。

## 13. 将来の方向性（v1完了後の候補。約束ではない）

- 愚門で検証したプロセスの、Agent / Skill / Command としての段階的な実装
- 検証済みMotion・Transition・Scene部品のライブラリ化（体験の組み合わせは案件固有のまま）
- 対応業種・対応フレームワークの段階的拡大
- 世界の最新Web表現の研究を継続的に取り込むリサーチプロセスの定型化
- Bloom-L の他基盤（CMS等）との連携
- Creative QA の観点拡充と、訪問者反応（滞在・到達率等）による事後検証

これらはすべて、v1の実案件検証で得た学びを反映してから着手する。
