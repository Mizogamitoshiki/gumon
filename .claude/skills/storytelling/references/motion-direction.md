# motion-direction — Stage 7の実行手順

目的: Motion一覧表とトークン定義を作る（`experience-plan-template.md` 8〜9章）。**設計のみ。実装・パッケージ追加はStage 9承認後**。
判断基準の正は `docs/creative/CREATIVE_DIRECTOR_ENGINE.md` 6章。技法の採用条件・禁忌は `docs/creative/KNOWLEDGE_SYSTEM.md` 5章。

## 7-1. Motion一覧表の作成

1. 全Sceneについて候補Motionを列挙し、表に書く。**表にないMotionは以降のStageで実装しない**（実装中の思いつき追加を禁止）
2. 各候補を判定する（順番厳守）:
   - **理由**: 憲章7.4の理由リスト（理解を助ける／期待を生む／感情を変える／世界観を強める／優先順位を示す／Sceneをつなぐ／信頼・行動へ導く）のどれかに、そのMotion固有の言葉で着地するか。「雰囲気が良くなる」は理由ではない → 行を削除
   - **下限**: 読解・操作を妨げないか。reduced-motion代替が設計できるか（できなければ削除）。性能予算内か
   - **単調**: 同型演出の連続≤2。全要素同型Reveal（量産感チェック第1項）になっていないか
   - **強度**: Rhythm Scoreの区間に従う。「静けさ・余韻・集中」区間では弱く。強化はピーク×記憶層×見せ場1〜2箇所×下限内のときのみ
3. **reduced-motion代替とモバイル代替は、この段階で同時に設計する**。後付けにしない
4. 削除判断条件（実装後にこのMotionを消す条件）を各行に書いておく

## 7-2. トークン定義

duration・easing・距離を場当たりでなくトークンセットとして定義する。水準はBriefのCinematic direction（ブランドトーン）から導く。

## 7-3. 技術選定の規則

**流行・見栄えだけで採用しない。** 各技術に採用理由を記録する。

| 技術 | 採用してよい条件 |
| --- | --- |
| CSS（transition/animation/scroll-driven） | 第一候補。単純なReveal・状態変化はまずCSSで足りるか検討する |
| GSAP + ScrollTrigger | タイムライン制御・scrub・pin等、CSSで保守的に書けない演出が設計に実在するとき |
| Lenis（スムーススクロール） | 慣性がCinematic directionに必要で、操作性・a11y・既存スクロール挙動への影響を評価済みのとき。「なめらかだから」だけでは入れない |
| SVG / Canvas | 図形・線・描画そのものが表現に必要なとき |
| Three.js / WebGL | 体験価値を大きく高め、DOM・CSS・GSAPで**代替しにくい**場合のみ（憲章8章）。採用時は個別のJS予算を記録 |

- 既存プロジェクトに同等の手段（framer-motion等）が既にあるなら、置き換えではなく既存を優先して検討する（既存スタックの尊重）
- 依存追加はすべてTransformation Planに列挙し、承認を得てから導入する。ライセンスも同時に確認する
- アニメーションはtransform / opacity中心。スクロール連動でのlayoutプロパティ操作をしない（QUALITY_STANDARD 16章）

## 完了条件

- Motion一覧表の全行に理由・代替・削除条件が記入済み（理由記録率100%）
- トークン定義がある
- 候補技術それぞれに採用理由がある（不要な技術は「不採用」と理由を記録）
