# implementation-gates — Stage 8〜11の実行手順

目的: 変換計画の提示 → 承認 → 代表Scene → 段階実装を、ゲートを飛ばさずに進める（`implementation-plan-template.md`使用）。

## Stage 8: Transformation Plan

設計成果物を、ユーザーが判断できる1つの計画に統合して提示する。含めるもの（テンプレート1章）:
現状分析の要約／残す・変更・追加・削除の4区分／Creative Concept／設計要約（感情曲線・Story・Scene map・Rhythm・Motion plan）／技術変更と依存パッケージ（理由・ライセンス付き）／リスクと低減策／実装順序／代表Sceneの選定と理由／QA方法。

提示の規則:
- **不利な材料を先に**: Assumptions（推定で進めた点）、Open Questions、リスク、実施できないQAを明示する
- 推奨と理由を添える（選択肢の丸投げをしない）
- 「削除するもの」にコンテンツが含まれる場合は特に目立たせる（コンテンツ削除は承認必須事項）

## Stage 9: Approval Gate（G1）

**承認前に行ってはならないこと**: 大規模な構造変更／依存パッケージの追加・インストール／コンテンツ削除／全面リデザイン／GSAP・Lenis等の導入／複数Sceneの一括実装／既存資産（画像・フォント・コンポーネント）の置換／新規サイトの場合のプロジェクトscaffold作成・フレームワーク選定の確定（Transformation Planの「技術変更」として提示し、承認を得る）。

承認の扱い:
- 承認は**範囲**として記録する（テンプレート2章）: 承認されたScene・ファイル・依存を文字で固定し、「承認されていない事項」も書く
- **曖昧な承認を全面実装の承認として扱わない。** 「いいね」「進めて」だけの場合、どの範囲への承認かを確認してから進む
- 計画への修正要望が出たら、該当Stageに戻って成果物を更新してから再提示する
- 以降のセッションでは、毎回この承認記録を読み直し、範囲外の作業をしない

## Stage 10: Representative Scene（G2）

1. Heroまたは代表Scene（記憶層Sceneが望ましい）**だけ**を実装する
2. 依存追加が承認範囲に含まれる場合も、この段階では代表Sceneに必要な最小限のみ導入する
3. 実装後、次を自ら確認して記録する: ブランド適合（Briefの根拠で説明できるか）／Motionの質感（トークン通りか）／Scroll rhythm（譜面通りか）／Typography／Spacing／Mobile（実機または375px）／reduced-motion／Performance（体感とDevTools）／Maintainability（Scene単位のモジュール、cleanup）
4. 結果を判定材料として整理し、**人間のCreative Reviewを依頼する**。合格するまで全面実装へ進まない
5. 不合格時は実装(コード)ではなく設計（Stage 4〜6）を先に疑う。原因Stageへ戻り、影響範囲を再検証する

## Stage 11: Incremental Implementation

- 承認済みSceneを**1Sceneずつ**（または結合度の高い最小単位で）実装する。一括の大規模書き換えは禁止
- 実装はScene単位でモジュール化し、一枚岩のアニメーションコードにしない（QUALITY_STANDARD 19章）
- **各Scene実装後のチェック**（テンプレート4章の行を埋める）:
  1. 設計との対応（Scene名・目的・強度が設計書と一致）
  2. 前後SceneとのTransition（境界を実際に低速で通過して確認）
  3. ScrollTrigger等のcleanup（アンマウント・リサイズ・ページ遷移でリーク／二重登録しない）
  4. レスポンシブ（375px〜。横スクロール発生なし）
  5. reduced-motionでコンテンツが全て見える（opacity:0待機の放置は禁止）
  6. コンソールエラーなし
  7. パフォーマンス悪化なし（体感＋long taskの目視）
  8. 既存機能（フォーム・ナビ・他ページ）への影響なし
- チェックに失敗したまま次のSceneへ進まない
