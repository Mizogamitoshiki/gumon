# project-analysis — Stage 1の実行手順

目的: 対象プロジェクトの現状を、判断せずに正確に観察し、`project-analysis-template.md` に記録する。

## 姿勢

- **観察と評価を分ける。** この段階で「ダサい」「使える」「作り直すべき」を言わない。評価は設計Stageの仕事であり、早い評価は観察を歪める
- 全記述に `[確認済み]/[推定]/[未確認]` を付ける。推測で埋めない
- **既存プロジェクトの尊重**: 現在の技術スタック・コンテンツ・運用は変換対象であって敵ではない。「残すべきもの」を先に探す

## 調査手順（既存サイト）

1. **技術**: package.json・設定ファイル・lockファイルから、フレームワーク／言語／スタイリング方式／ルーティング／ビルド・実行・テスト方法／既存アニメーション依存（gsap, framer-motion, lenis等）を特定
2. **構成**: ページ一覧と役割、主要Component、トップページのセクション構成
3. **アセット**: 画像・映像・フォントの所在と品質。**ライセンスの手がかり**（フォント名・素材の出所）を必ず記録
4. **現在の体験**: 実装済みアニメーションの種類・場所・実装方法。CTAの種類・位置・動作。可能なら実際に起動して等速で通しスクロールし、所感を「観察」として記録
5. **ブランドの手がかり**: 既存コピー・About・こだわりページ等から、クライアント自身の言葉を**原文のまま**採取（要約しない。Interviewとコピーの原料になる）
6. **懸念**: パフォーマンス（巨大画像・重い依存・同期読み込み）、アクセシビリティ（コントラスト・alt・キーボード）、変更リスク（壊してはいけない導線・フォーム・SEO資産・外部連携）
7. **情報ギャップ一覧**: テンプレート8章に、[未確認]事項と「表現判断への影響」「解析で埋まるか」を記入 → Stage 2の入力

## 新規サイトの場合（コードがほぼ無い）

Stage 1をスキップしない。調査対象をコードから**材料**に切り替える:

- クライアント提供資料（写真・メニュー・パンフ・SNS・既存の紹介記事）を同じ姿勢で観察する
- 技術スタックは「制約」として扱う（指定があるか、ホスティング・運用体制はどうか）
- テンプレートの該当しない欄は「該当なし（新規）」と明記し、空欄のまま放置しない

## 完了条件

- テンプレートの全欄が記入済み（「該当なし」含む）
- ラベルなしの記述がない
- 情報ギャップ一覧が書けている
- 保存場所（既定: 対象プロジェクトの`docs/cinematic/project-analysis.md`）に保存済み
