# qa-protocol — Stage 12〜14の実行手順

目的: Technical QA → Creative QA → 完了報告を、正直に実施する（`qa-report-template.md`使用）。
合否基準の正は `docs/QUALITY_STANDARD.md`（Technical=22章・Creative=21章・各項目=1〜20章）。

## 大原則

- **Technical QAが先。** 技術的に壊れた状態でCreative QAをしても意味がない
- **実行環境・ツールがない検証を実施済みとして扱わない。** 未実施は「未実施」と理由付きで書く。これはSkillの失敗ではない — 偽装が失敗である
- パフォーマンスのラボ計測はフィールド（実ユーザー）を保証しない。報告では**「ラボ計測・フィールド未計測」**と正確に書く（QUALITY_STANDARD 16章）

## Stage 12: Technical QA

qa-report-template 1章の全項目を実施または「未実施＋理由」で埋める:
Build／Test／Lint／Type check／コンソールエラー（本番ビルド）／アクセシビリティ自動チェック／reduced-motion切替の通し確認／レスポンシブ（375・768・1024・1440・1920）／対象ブラウザ（可能な範囲。確認できない環境は未実施と書く）／CTA動作（送信・電話・予約導線の実測）／パフォーマンス（Lighthouseモバイル・CPU 4倍スロットリング通し）／Animation cleanup／依存・素材・フォントのライセンス。

- Blocker不合格が出たら修正し、**修正が影響する項目のみ**再QAする
- ブラウザ実機など環境がない項目は、人間による確認が必要な項目（テンプレート3章）へ回す

## Stage 13: Creative QA

qa-report-template 2章の全観点について**判定材料を整理する**。判定の主体は人間であり、Claudeは感性品質を断定しない。

実施方法:
1. **通し体験**: PC・モバイルで等速スクロールを最後まで。好奇心が切れた地点・疲れた地点・記憶に残った箇所を時系列で記録
2. **設計照合**: 感情曲線・Story一文・Scene定義・譜面・Motion一覧と実装を突き合わせ、乖離を列挙
3. **削除テスト**: 主要Motionを無効化して比較。「無い方が良い／変わらない」ものは削除する
4. **量産感チェック**（QUALITY_STANDARD 9章の5項目）: 全要素同型出現／汎用構成の踏襲／デフォルト配色・紫グラデ／汎用文言だけのコピー／業種名を入れ替えても成立 — を1項目ずつ判定
5. **疲労チェック**: 3回連続閲覧の所感
6. 第三者テスト・「このブランドらしいか」の最終判定・ボーダーライン判断は、**人間による確認が必要な項目**として明示的にリスト化する

## Stage 14: Completion Report

報告に含めるもの（この順で）:
1. 実施した内容（Stage単位の要約）
2. 変更ファイル一覧
3. 主要な設計判断とその根拠（Briefへの参照）
4. 実行した検証と結果（合格項目）
5. **不合格・未検証の項目**（隠さない。不利な材料を先に）
6. 人間による確認が必要な項目
7. 残るリスク
8. 次の改善候補（公開後のフィールド計測の案内を含む: CrUX/RUMでLCP・CLS・INPを確認し運用QAとして記録する）

成果物（qa-report.md）を保存し、各成果物のステータスを更新して終了する。
